## Abstract

Large Language Models operate within fixed context windows, a fundamental architectural constraint that limits their ability to maintain coherent long-running conversations. As dialogue length increases, earlier context is silently evicted, causing models to lose track of user preferences, established facts, and prior decision-making rationale. This paper presents Deer, a memory architecture that addresses the context window limitation through hierarchical compression. The system employs a three-tier approach: raw recent turns for immediate attention, structured memory slots for key facts, and a KVTC/TurboQuant encoded archive for long-term storage. Benchmarking across 50–500 turn conversations demonstrates compression ratios of 82–98% (compared to 73–95% for raw gzip), with sub-millisecond latency for typical workloads.

---

## 1. Introduction

The Transformer architecture, despite its remarkable capabilities, suffers from a well-documented limitation: fixed attention windows. Claude, GPT, Grok, and their contemporaries all operate within bounded context, typically ranging from 4K to 128K tokens depending on the model variant. In practical deployment, users engaging in extended conversations rapidly exceed these limits.

The consequence is subtle but significant. When context overflow occurs, older messages are not flagged or summarized—they simply cease to exist from the model's perspective. A user who spent thirty minutes establishing project requirements, specifying coding preferences, and defining architectural decisions will find all of that context lost once the window fills. The model responds as if the conversation began moments ago.

This problem motivated Deer. Rather than attempting to extend the model's context window directly, which requires architectural changes to the model itself, Deer operates as an external memory layer. It sits between the user and the model, managing the conversation state in a way that allows retrieval of relevant context when needed.

## 2. System Architecture

### 2.1 Hierarchical Memory Model

Deer implements three distinct memory layers, each serving a different purpose in the retrieval hierarchy.

The first layer consists of recent turns kept in their raw JSON format. These messages remain unprocessed to preserve all semantic detail for immediate attention. The system maintains up to thirty recent turns in this form, a number chosen to balance context relevance against token overhead.

The second layer comprises memory slots, structured containers generated by the language model itself. These slots capture high-level facts—user preferences, project state, key decisions—rather than raw conversation. The schema for these slots is model-generated and context-dependent.

The third layer is the coded archive, where long-term memory lives in compressed form. This archive is stored in a field called __deflate within the JSON archive, the product of the KVTC/TurboQuant pipeline described below.

### 2.2 Compression Pipeline

The compression system transforms raw conversation into the __deflate field through a multi-stage process.

**Chunking.** Conversation turns are grouped into chunks of eight. This size was determined empirically, balancing the number of features extracted against the granularity of retrieval.

**Feature Extraction.** Each chunk undergoes analysis to extract numeric features: total character length normalized against a denominator, word count normalized similarly, boolean indicators for code presence, tool or function references, decision language, and text diversity ratio for non-fast compression levels.

**KVTC Stage.** The feature matrix undergoes Singular Value Decomposition using Eigen's JacobiSVD implementation. SVD reduces dimensionality by projecting the feature space onto its principal components, capturing the most significant variance while discarding noise.

**TurboQuant Stage.** Following dimensionality reduction, the coefficients undergo random orthogonal rotation. A random orthogonal matrix is computed via QR decomposition of a random matrix, then applied to rotate the coefficient space. This rotation redistributes information in a way that improves compressibility without altering the underlying structure.

**Quantization.** The rotated coefficients are quantized to four-bit resolution. Values are mapped to the range [-8, +7] by scaling by 15.0 and rounding. This aggressive quantization reduces each feature to a single nibble.

**Deflate and Encoding.** The quantized matrix is serialized to JSON, then compressed using zlib's deflate algorithm. The binary result is encoded as hexadecimal to ensure safe storage within JSON, avoiding UTF-8 complications that arise from raw binary data.

The complete pipeline operates in under one millisecond for typical conversation volumes.

## 3. Implementation

### 3.1 Code Organization

The implementation follows a Layer-based architectural pattern with dependency injection. The system uses layer composition to build services: config, then logger/storage/compress, then cli. Each layer is a composable unit with explicit dependencies.

The core headers reside in include/deer: types/config.h defines compression levels and constants, types/archive.h contains the ArchiveState structure, types/errors.h specifies error types, types/version.h declares version strings, and types/ contains service interfaces (cli_service.h, compress_service.h, logger.h, storage.h). The layer implementations in layers/ (config_layer.h, logger_layer.h, storage_layer.h, compress_layer.h, cli_layer.h) provide the concrete service construction. The operational layer in ops/ provides archive loading and saving in ops/archive.h, while ops/compress.h implements the compression and decompression logic. The CLI layer in cli/cli.h handles command routing.

### 3.2 Compression Levels

The system supports three compression levels distinguished by feature count.

Fast compression extracts twelve features per chunk and executes most quickly. Balanced, the default, extracts twenty features. Maximum compression extracts twenty-eight features, providing the finest granularity at the cost of slightly longer execution time.

### 3.3 Dependencies Analysis

Deer relies on three external libraries, each selected for specific technical reasons aligned with the system's requirements.

**Eigen** serves as the linear algebra backend. The KVTC compression stage requires Singular Value Decomposition to reduce the feature matrix dimensionality. Eigen's JacobiSVD implementation provides robust SVD computation with full V matrix output needed for coefficient projection. The library header-only nature simplifies distribution, and its templated architecture ensures zero-cost abstraction at compile time. Eigen is the de facto standard for C++ linear algebra, with mature GPU support should future versions require CUDA acceleration.

The choice of Eigen reflects the SVD-based nature of KVTC compression. Alternative approaches could use LAPACK directly, but Eigen's C++ interface provides better type safety and integration with the STL containers used throughout the codebase.

**nlohmann-json** handles all JSON serialization and deserialization. Deer stores conversation state, compressed archives, and configuration in JSON format for portability and human readability. This library was selected over alternatives for several reasons: header-only distribution matching Eigen's model, constexpr parsing for compile-time validation, and an intuitive chainable API that mirrors C++ stream semantics. The library's exception safety guarantees align with Deer's error-handling philosophy.

JSON serialization is fundamental to Deer's design. The archive format must be portable across systems and recoverable without specialized tools. While binary formats could achieve marginally better compression, JSON provides the necessary debuggability and interoperability.

**zlib** provides the deflate compression algorithm. The TurboQuant pipeline concludes with zlib compression to eliminate redundancy in the quantized coefficient matrix. The library's proven stability and ubiquitous availability make it the default choice for compression. Deer's usage pattern—compressing moderate-sized JSON strings rather than streaming data—aligns well with zlib's synchronous API.

The choice of zlib over alternatives like LZ4 or Zstandard reflects the specific characteristics of Deer's data. After quantization, the coefficient matrix contains highly redundant patterns that deflate handles effectively. LZ4 would prioritize speed but sacrifice compression ratio; Zstandard offers a middle ground but introduces additional complexity. For the sub-millisecond operation times Deer's target workload demands, zlib's compression efficiency proved sufficient.

These three dependencies together provide the mathematical, serialization, and compression capabilities necessary for the KVTC/TurboQuant pipeline. Each was selected to balance performance, portability, and maintainability.

All experiments were run with seeded PRNG (seed=42) for reproducibility. Conversation data is synthetically generated from a pool of varied dialogue templates to provide controlled conditions. Build and run `deer_bench` to reproduce (see Section 4.5).

## 5. Usage

### 5.1 Building

Deer requires Eigen3, nlohmann-json, and zlib. On macOS, install these via Homebrew. On Ubuntu, apt-get install the eigen3, nlohmann-json3-dev, and zlib1g-dev packages.

Build with CMake:

```bash
mkdir -p build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
cmake --build .
```

The compiled binary appears at build/deer. Optionally copy it to a location in your PATH.

### 5.2 Commands

Add messages to the archive by piping JSON to stdin:
```bash
echo '{"content": "My name is Chris", "role": "user"}' | deer add
```

View the current archive state with resume:
```bash
deer resume
```

Compress with any of the three levels:
```bash
deer compress --fast
deer compress --balanced
deer compress --max
```

Decompress to inspect the restored feature matrix:
```bash
deer decompress
```

The help command displays usage information.

## 6. Archive Format

### 6.1 JSON Structure

The archive file `deer_archive.json` contains the complete memory state:

```json
{
  "codedArchive": {
    "__deflate": "789c2589c111003008c2169207da4ee...",
    "original_size": 71,
    "compressed_size": 33,
    "encoding": "hex",
    "method": "deer_v0.6_research",
    "chunks": 1,
    "level": "max"
  },
  "recentTurns": [
    {
      "content": "My name is Chris",
      "role": "user"
    },
    {
      "content": "Hello Chris, nice to meet you",
      "role": "assistant"
    }
  ],
  "memorySlots": [],
  "tokenEstimate": 42
}
```

### 6.2 Field Definitions

| Field | Type | Description |
|-------|------|-------------|
| `codedArchive` | object | Compressed long-term memory storage |
| `__deflate` | string | Hex-encoded compressed feature matrix |
| `original_size` | integer | Size of feature matrix before compression (bytes) |
| `compressed_size` | integer | Size after zlib deflate (bytes) |
| `encoding` | string | Encoding method, always "hex" |
| `method` | string | Compression algorithm version |
| `chunks` | integer | Number of conversation chunks processed |
| `level` | string | Compression level used (fast/balanced/max) |
| `recentTurns` | array | Raw conversation turns for immediate attention |
| `memorySlots` | array | Structured facts extracted by LLM |
| `tokenEstimate` | integer | Estimated token count of stored content |

### 6.3 Resume Output

When running `deer resume`, the output is formatted for LLM consumption:

```json
{
  "archive": {
    "coded": {
      "__deflate": "789c2589c111003008c2169207da4ee...",
      "chunks": 1,
      "level": "max"
    },
    "recent": [
      {"content": "My name is Chris", "role": "user"},
      {"content": "Hello Chris", "role": "assistant"}
    ],
    "slots": []
  },
  "system": "You are resuming from the Deer memory archive."
}
```

## 7. Conclusion

Deer demonstrates that hierarchical compression can dramatically reduce the storage requirements of conversation history. The KVTC/TurboQuant pipeline achieves 82–98% compression across 50–500 turn conversations (outperforming raw gzip by 8–17 percentage points), with sub-millisecond latency for conversations up to 200 turns. The aggressive INT4 quantization introduces significant point-wise distortion in the feature space, which is an intentional tradeoff: the compressed archive serves as a structural fingerprint rather than a lossless store, complementing the uncompressed recent turns and structured memory slots that carry the semantic load. The system provides a practical solution to the context window limitation that does not require modifications to the underlying language model itself.

The architecture is available under the [MIT license](https://github.com/chrismichaelps/deer/blob/main/LICENSE). The implementation uses modern C++20 with Layer-based dependency injection architecture. Eigen handles linear algebra, nlohmann-json handles serialization, and zlib provides deflate compression. All evaluation results are reproducible via the `deer_bench` harness (see Section 4.5).
